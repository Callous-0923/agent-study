==================================================

 

 

 

DSPy: Stanford NLP 实验室，github.com/stanfordnlp/dspy

核心理念来自论文: "DSPy: Compiling Declarative LM Calls into Self-Improving Pipelines"

 

 

22.1 编程 vs 提示词 —— DSPy 的设计哲学

 

传统 Prompt 工程的问题：

 

你写了一个 Agent：

prompt = "你是一个助手，请用中文回答..."

 

换一个模型（GPT-4 → Claude）：

→ Prompt 要重新调整
→ 效果可能大幅下降
→ 这就是「Prompt 脆弱性」

 

DSPy 的解决方案：

把 Prompt 当作「代码的编译产物」而非手写字符串

 

类比：

传统 Prompt = 手动写汇编代码（每换一个 CPU 就要重写）
DSPy = 写 C 代码再编译（代码不变，编译器适配不同 CPU）

 

翻译成 DSPy 术语：

DSPy 程序（Python 代码）→ Optimizer（编译器）→ Prompt（机器码）
                                       ↑
                                自动适配不同的 LLM

 

 

22.2 Signature —— 把 Prompt 变成「函数签名」

 

传统 Prompt：

prompt = "给定问题 {question}，请给出答案" # 脆弱的字符串！

 

DSPy Signature：

class QA(dspy.Signature):

  question = dspy.InputField(desc="问题")
  answer = dspy.OutputField(desc="答案")

 

Signature 的优势：

  • 声明式：描述「输入什么、输出什么」，不碰字符串
  • 自文档化：desc 即文档，人类和 Optimizer 都能理解
  • 可组合：Signature 可以拼接成复杂流水线
  • 可验证：输出类型自动校验

    22.3 Module —— 把 Agent 组件变成「函数」

     

    DSPy Module 是可组合的 AI 组件。

     

    内置 Module 类型：

     

      ┌──────────────────┬──────────────────────────────────┐
  │ Module            │        作用                       │
  ├──────────────────┼──────────────────────────────────┤
  │ dspy.Predict      │ 最基本的单步 LLM 调用             │
  │ dspy.ChainOfThought│ 带推理链的 LM 调用               │
  │ dspy.ReAct        │ ReAct Agent 循环                 │
  │ dspy.MultiChain   │ 多链并行 + 投票                  │
  │ dspy.ProgramOfThought│ 用代码辅助推理                │
  └──────────────────┴──────────────────────────────────┘

     

    类比 LangChain：

    LangChain Chain ≈ DSPy Module

    区别：DSPy Module 的参数（prompts/weights）可由 Optimizer 自动调优

     

    Agent 开发者用 DSPy 的样子：

     

    class AgentQA(dspy.Module):

      def __init__(self):
      self.search = dspy.ChainOfThought("query -> results")
      self.answer = dspy.ChainOfThought("context, question -> answer")

     

      def forward(self, question):
      results = self.search(query=question)
      return self.answer(context=results, question=question)

    22.4 Optimizer (Teleprompter) —— 自动优化 Prompt

     

    这是 DSPy 最革命性的部分。

     

    传统做法（手调 Prompt）：

  • 写一个 Prompt
  • 在 5 个例子上测试
  • 效果不好 → 修改 Prompt → 再测试
  • 换模型 → 全部重来

    •  

    DSPy 做法（自动优化）：

  • 定义 Metric（评分函数）
  • 准备训练集（10 个标注样本就够）
  • 运行 Optimizer → 自动调出最优 Prompt

    •  

    DSPy 的 Optimizer 类型：

     

      ┌─────────────────────┬──────────────────────────────────┐
  │ Optimizer            │         做什么                    │
  ├─────────────────────┼──────────────────────────────────┤
  │ BootstrapFewShot    │ 自动找出最佳少样本示例              │
  │ MIPROv2             │ 自动探索最优指令 + 示例组合         │
  │ BootstrapFinetune   │ 为每个 Module 自动微调小模型        │
  │ BetterTogether      │ 组合多个 Optimizer 级联优化         │
  └─────────────────────┴──────────────────────────────────┘

     

    工作流程（MIPROv2 为例）：

     

  • Bootstrapping 阶段：
    •  运行未优化的程序多次 → 收集成功轨迹
 → 筛选出得分高的输入输出对 → 作为候选示例

     

  • Grounded Proposal 阶段：
    •  分析程序代码 + 数据 + 轨迹
 → 生成多个候选指令（instruction variants）

     

  • Discrete Search 阶段：
    •  在训练集上尝试不同「指令 + 示例」的组合
 → 用 Metric 评分 → 选出最优组合

    22.5 DSPy + Agent 的实战价值

     

    DSPy 不是 LangChain 的替代品，是补充：

     

    DSPy 擅长：

    ✓ 自动优化 Agent 每一步的 Prompt
✓ 发现最佳的 few-shot 示例
✓ 结构化输出的质量保证

     

    LangChain 擅长：

    ✓ Agent 编排（LangGraph 的图式工作流）
✓ 工具集成（MCP、向量数据库、API）
✓ 生态系统（社区、文档、第三方集成）

     

    两者配合：

    用 LangGraph 搭建 Agent 的执行流程
用 DSPy 优化每个 LLM 调用节点的 Prompt

     

    面试中这样描述会非常加分：

    "我们用 LangGraph 做 Agent 编排（控制流），

    DSPy 做 LLM 调用优化（Prompt 层），

    两者解耦，各司其职。"

     

     

    22.6 DSPy 把 Agent 开发变成「数据驱动」

     

    传统 Agent 开发流程：

    Prompt 不好 → 看结果 → 凭直觉改 → 再试

     

    DSPy 的 Agent 开发流程：

    写代码（Module）→ 准备评测集 → 运行 Optimizer → 自动优化完成

     

    类比：

    传统方式 = 手动调参炼丹

    DSPy = 自动超参搜索（类似 Optuna 之于深度学习）

     

    面试金句：

    "我不手动调 Prompt，我写评测指标，

    让 DSPy Optimizer 自动找出最佳 Prompt。"

    22.7 本章总结

     

    核心要点回顾：

     

  • DSPy = Programming, Not Prompting
  • Prompt 是编译产物，不是手写字符串

    •

"n">22.1 编程 vs 提示词 —— DSPy 的设计哲学
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

传统 Prompt 工程的问题：

  你写了一个 Agent：
    prompt = "你是一个助手，请用中文回答..."

  换一个模型（GPT-"n">4 → Claude）：
    → Prompt 要重新调整
    → 效果可能大幅下降
    → 这就是「Prompt 脆弱性」

DSPy 的解决方案：
  把 Prompt 当作「代码的编译产物」而非手写字符串

  类比：
    传统 Prompt = 手动写汇编代码（每换一个 CPU 就要重写）
    DSPy = 写 C 代码再编译（代码不变，编译器适配不同 CPU）

  翻译成 DSPy 术语：
    DSPy 程序（Python 代码）→ Optimizer（编译器）→ Prompt（机器码）
                                           ↑
                                    自动适配不同的 LLM


"n">22.2 Signature —— 把 Prompt 变成「函数签名」
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

传统 Prompt：
  prompt = "给定问题 {question}，请给出答案"  "k">class="h"># 脆弱的字符串！

DSPy Signature：
  "k">class "f">QA(dspy.Signature):
      question = dspy.InputField(desc="问题")
      answer = dspy.OutputField(desc="答案")

Signature 的优势：
  "n">1. 声明式：描述「输入什么、输出什么」，不碰字符串
  "n">2. 自文档化：desc 即文档，人类和 Optimizer 都能理解
  "n">3. 可组合：Signature 可以拼接成复杂流水线
  "n">4. 可验证：输出类型自动校验

"n">22.3 Module —— 把 Agent 组件变成「函数」
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

DSPy Module 是可组合的 AI 组件。

内置 Module 类型：

  ┌──────────────────┬──────────────────────────────────┐
  │ Module            │        作用                       │
  ├──────────────────┼──────────────────────────────────┤
  │ dspy.Predict      │ 最基本的单步 LLM 调用             │
  │ dspy.ChainOfThought│ 带推理链的 LM 调用               │
  │ dspy.ReAct        │ ReAct Agent 循环                 │
  │ dspy.MultiChain   │ 多链并行 + 投票                  │
  │ dspy.ProgramOfThought│ 用代码辅助推理                │
  └──────────────────┴──────────────────────────────────┘

类比 LangChain：
  LangChain Chain ≈ DSPy Module
  区别：DSPy Module 的参数（prompts/weights）可由 Optimizer 自动调优

Agent 开发者用 DSPy 的样子：

  "k">class "f">AgentQA(dspy.Module):
      "k">def "f">__init__("k">self):
          "k">self.search = dspy.ChainOfThought("query -> results")
          "k">self.answer = dspy.ChainOfThought("context, question -> answer")

      "k">def "f">forward("k">self, question):
          results = "k">self.search(query=question)
          "k">return "k">self.answer(context=results, question=question)
"""


"k">class "f">DSPyModule:

        "k">self.prompt = signature.to_prompt_template()
        "k">self.optimized_prompt = "k">self.prompt  "k">class="h"># Optimizer 会修改这个值
        "k">self.few_shot_examples = []  "k">class="h"># Optimizer 会填充这个列表

    "k">def "f">__call__("k">self, **inputs) -> "b">dict:
        "k">class="s">"""执行模块（模拟 LLM 调用）。"""
        "k">class="h"># 组装最终 prompt

                "k">for ex "k">in "k">self.few_shot_examples[-"n">3:]

                         "k">if v["role"] == "output"]

"n">22.4 Optimizer (Teleprompter) —— 自动优化 Prompt
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

这是 DSPy 最革命性的部分。

传统做法（手调 Prompt）：
  "n">1. 写一个 Prompt
  "n">2. 在 "n">5 个例子上测试
  "n">3. 效果不好 → 修改 Prompt → 再测试
  "n">4. 换模型 → 全部重来

DSPy 做法（自动优化）：
  "n">1. 定义 Metric（评分函数）
  "n">2. 准备训练集（"n">10 个标注样本就够）
  "n">3. 运行 Optimizer → 自动调出最优 Prompt

DSPy 的 Optimizer 类型：

  ┌─────────────────────┬──────────────────────────────────┐
  │ Optimizer            │         做什么                    │
  ├─────────────────────┼──────────────────────────────────┤
  │ BootstrapFewShot    │ 自动找出最佳少样本示例              │
  │ MIPROv2             │ 自动探索最优指令 + 示例组合         │
  │ BootstrapFinetune   │ 为每个 Module 自动微调小模型        │
  │ BetterTogether      │ 组合多个 Optimizer 级联优化         │
  └─────────────────────┴──────────────────────────────────┘

工作流程（MIPROv2 为例）：

  "n">1. Bootstrapping 阶段：
     运行未优化的程序多次 → 收集成功轨迹
     → 筛选出得分高的输入输出对 → 作为候选示例

  "n">2. Grounded Proposal 阶段：
     分析程序代码 + 数据 + 轨迹
     → 生成多个候选指令（instruction variants）

  "n">3. Discrete Search 阶段：
     在训练集上尝试不同「指令 + 示例」的组合
     → 用 Metric 评分 → 选出最优组合
"""


"k">class "f">DSPyOptimizer:
    "k">class="s">"""模拟 DSPy Optimizer —— 演示自动优化的工作流。

    三样东西：
      "n">1. 你的 DSPy 程序（Module）
      "n">2. Metric（评分函数）
      "n">3. 训练输入（少量带标注的样本）
    """

    "k">def "f">__init__("k">self, metric: "b">callable):
        "k">class="s">"""
        Args:
            metric: 评分函数: (predicted, expected) -> score ("n">0-"n">1)
        """
        "k">self.metric = metric
        "k">self.candidates = []

    "k">def "f">compile("k">self, module: DSPyModule,
                trainset: "b">list["b">dict],
                max_candidates: "b">int = "n">5) -> DSPyModule:
        "k">class="s">"""编译（优化）一个 Module。

        流程：
          "n">1. 收集轨迹（多次运行，记录输入输出）
          "n">2. 筛选高分轨迹
          "n">3. 生成候选 Prompt 变体
          "n">4. 在训练集上评估，选最优

        Args:
            module: 要优化的 Module。
            trainset: 训练数据（输入+期望输出）。
            max_candidates: 生成的候选数。

        Returns:
            优化后的 Module。
        """
        "k">class="h"># 阶段 "n">1：Bootstrapping（收集成功案例）
        trajectories = []
        "k">for sample "k">in trainset:
            inputs = sample.get("inputs", {})
            expected = sample.get("outputs", {})
            result = module(**inputs)
            score = "k">self.metric(result, expected)
            "k">if score > "n">0.6:
                trajectories.append({
                    "inputs": inputs,
                    "outputs": result,
                    "expected": expected,
                    "score": score,
                })

        "k">class="h"># 阶段 "n">2：筛选 best-shot 示例
        trajectories.sort(key="k">lambda t: t["score"], reverse="k">True)
        best_trajectories = trajectories[:max_candidates]

        "k">class="h"># 阶段 "n">3：注入优化结果
        module.few_shot_examples = [
            {"Q": t["inputs"], "A": t["outputs"]}
            "k">for t "k">in best_trajectories
        ]

        "k">class="h"># 阶段 "n">4：生成优化后的指令（简化版）
        "k">if best_trajectories:
            avg_score = "b">sum(t["score"] "k">for t "k">in best_trajectories) / "b">len(best_trajectories)
            module.optimized_prompt = (
                module.signature.to_prompt_template() +
                f"\n\n[优化提示: 基于 {">len(best_trajectories)} 个样本调优, "
                f"平均得分 {avg_score:.">1%}]"
            )

        "k">self.candidates = best_trajectories
        "k">return module

"k">def "f">make_metric(expected_key: "b">str):
    "k">class="s">"""创建评分函数。"""
    "k">def "f">metric(predicted: "b">dict, expected: "b">dict) -> "b">float:
        pred_val = "b">str(predicted.get(expected_key, ""))
        exp_val = "b">str(expected.get(expected_key, ""))
        "k">if pred_val == exp_val:
            "k">return "n">1.0
        "k">if exp_val.lower() "k">in pred_val.lower():
            "k">return "n">0.7
        "k">class="h"># Token 重叠率
        pred_tokens = "b">set(pred_val.lower().split())
        exp_tokens = "b">set(exp_val.lower().split())
        "k">if "k">not exp_tokens:
            "k">return "n">0.0
        overlap = "b">len(pred_tokens & exp_tokens) / "b">len(exp_tokens)
        "k">return "b">min(overlap, "n">0.5)
    "k">return metric
