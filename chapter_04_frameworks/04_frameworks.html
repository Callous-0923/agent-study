4.1 LangChain Agent

 

LangChain 是最流行的 LLM 应用框架，它将第1-3章我们

手写的概念（Agent、Tool、Memory）都抽象为了标准组件。

 

核心组件映射：

手写版 → LangChain 版

───────────── ─────────────────

TOOLS 列表 → @tool 装饰器

TOOL_MAP 字典 → Tool 对象

Agent 循环 → create_react_agent()

对话历史 → MemorySaver / ChatMessageHistory

 

安装依赖:

pip install langchain langchain-openai langgraph

4.2 LangGraph —— 状态机 Agent

 

LangGraph 是 LangChain 生态中的 Agent 编排框架。

它的核心思想是用「有向图」来定义 Agent 的执行流程。

 

核心概念：

  • State: 贯穿整个流程的共享状态（TypedDict）
  • Node: 图中的一个执行节点（Python 函数）
  • Edge: 节点之间的连线（普通边 / 条件边）

    •  

    什么时候用 LangGraph 而不是 LangChain Agent？

  • 需要自定义执行流程（不是简单的 ReAct 循环）
  • 需要在步骤间传递复杂状态
  • 需要并行执行、条件分支、循环等复杂控制流
  • 多 Agent 协作场景（第5章会用到）

    •  

    安装依赖：

    pip install langgraph

"k">class="h"># 定义状态类型
"k">class "f">AgentState(TypedDict):
    "k">class="s">"""LangGraph Agent 的状态定义。

    这是 LangGraph 最核心的概念。所有节点共享这个状态，
    节点通过读取/修改状态来协作完成任务。

    Annotated["b">list, add_messages] 表示消息列表使用
    追加模式（新消息添加到末尾，不会覆盖）。
    """
    messages: Annotated["b">list, add_messages]
    task_result: "b">str
    step_count: "b">int


"k">def "f">demo_langgraph_agent():
    "k">class="s">"""演示 LangGraph 状态机 Agent。

    实现一个简单的分析流程：
      输入问题 → LLM 分析 → 判断是否需要工具 → 输出结论

    LangGraph 让我们可以自定义 Agent 的执行图结构，
    而不是拘泥于固定的 ReAct 循环。
    """
    "k">if "k">not LANGGRAPH_AVAILABLE "k">or "k">not LANGCHAIN_AVAILABLE:
        "b">print("需要安装 langgraph 和 langchain 才能运行此演示。")
        "k">return

    "b">print("\n" + "=" * "n">60)
    "b">print("  LangGraph 状态机 Agent 演示")
    "b">print("=" * "n">60)

    llm = ChatOpenAI(
        model=os.getenv("LLM_MODEL", "gpt-4o-mini"),
        api_key=os.getenv("OPENAI_API_KEY"),
        base_url=os.getenv("OPENAI_BASE_URL"),
        temperature="n">0.5,
    )

    "k">class="h"># ===== 定义节点函数 =====
    "k">def "f">analyze_node(state: AgentState) -> AgentState:
        "k">class="s">"""分析节点：让 LLM 理解用户输入。"""
        "b">print("  [Node: analyze] 正在分析用户输入...")
        response = llm.invoke([
            ("system", "分析用户问题，提取关键信息。简洁回答。"),
            ("user", state["messages"][-"n">1].content),
        ])
        state["messages"].append(response)
        state["task_result"] = response.content
        state["step_count"] = state.get("step_count", "n">0) + "n">1
        "k">return state

    "k">def "f">should_continue(state: AgentState) -> "b">str:
        "k">class="s">"""条件判断：是否需要进一步处理？"""
        result = state["task_result"]
        "k">if "b">len(result) < "n">50:
            "b">print("  [Edge] 回答太短，需要补充详细分析")
            "k">return "elaborate"
        "b">print("  [Edge] 回答充分，结束流程")
        "k">return "end"

    "k">def "f">elaborate_node(state: AgentState) -> AgentState:
        "k">class="s">"""扩展节点：补充详细分析。"""
        "b">print("  [Node: elaborate] 正在补充详细分析...")
        response = llm.invoke([
            ("system", "请更详细地展开分析，提供更多上下文和细节。"),
            ("user", state["task_result"]),
        ])
        state["messages"].append(response)
        state["task_result"] = response.content
        state["step_count"] = state.get("step_count", "n">0) + "n">1
        "k">return state

    "k">class="h"># ===== 构建图 =====
    workflow = StateGraph(AgentState)

    "k">class="h"># 添加节点
    workflow.add_node("analyze", analyze_node)
    workflow.add_node("elaborate", elaborate_node)

    "k">class="h"># 设置入口点
    workflow.set_entry_point("analyze")

    "k">class="h"># 添加条件边：从 analyze 根据条件走向不同分支
    workflow.add_conditional_edges(
        "analyze",
        should_continue,
        {
            "elaborate": "elaborate",
            "end": END,
        },
    )

    "k">class="h"># elaborate 完成后结束
    workflow.add_edge("elaborate", END)

    "k">class="h"># 编译图
    app = workflow.compile()

    "k">class="s">"""
    图结构示意：

      [START]
         │
         ▼
    ┌──────────┐     条件判断      ┌───────────┐
    │  analyze  ├──────────────────→│ elaborate  │
    └─────┬─────┘  (太短)          └─────┬─────┘
          │ (足够)                       │
          ▼                              ▼
       [END]                          [END]

    这就是 LangGraph 的核心优势：
    用代码定义复杂的执行流程，而非硬编码在循环中。
    """

    "k">class="h"># 运行
    test_input = "LangGraph 是什么？请简短说明。"
    "b">print(f"\n👤 用户: {test_input}")

    initial_state = {
        "messages": [HumanMessage(content=test_input)],
        "task_result": "",
        "step_count": "n">0,
    }

    final_state = app.invoke(initial_state)
    "b">print(f"\n🤖 Agent: {final_state['task_result']}")
    "b">print(f"   总步骤数: {final_state['step_count']}")

    4.3 框架选择策略（面试高频！）

     

    面试官：「你用过哪些 Agent 框架？怎么选型？」

     

    回答框架：

     

  • 学习阶段：从裸写开始（第1章）
  • 理解底层原理
  • 不被框架黑盒限制

    •  

  • 原型验证：用 LangChain Agent
  • 快速搭建、生态丰富
  • 适合 MVP 和 demo

    •  

  • 生产环境：用 LangGraph
  • 状态管理可控
  • 支持复杂工作流
  • 可观测性好（便于调试和监控）

    •  

  • 多 Agent 场景：
  • crewAI（简单协作）
  • AutoGen（需要人在回路）
  • 自建 LangGraph 图（完全定制）

    •  

    选型核心考虑因素：

      ┌──────────────┬────────────────────────────────────┐
  │    因素       │               说明                  │
  ├──────────────┼────────────────────────────────────┤
  │ 复杂度        │ 简单用 LangChain，复杂用 LangGraph  │
  │ 可控性        │ 生产环境优先 LangGraph              │
  │ 团队能力      │ 新手用 LangChain，熟手自建           │
  │ 成本          │ 裸写 < LangChain < LangGraph       │

    4.4 本章总结

     

    核心要点回顾：

     

  • LangChain Agent
  • create_react_agent() 一行创建 Agent
  • @tool 装饰器定义工具
  • MemorySaver 自动管理对话历史
  • 适合快速原型和中等复杂度场景

    •  

  • LangGraph
  • 状态机模型：State + Node + Edge
  • 完全自定义执行流程
  • 支持条件分支、循环、并行
  • 生产环境首选

    •  

  • 选型策略
  • 学习原理 → 裸写
  • 快速验证 → LangChain
  • 生产落地 → LangGraph
  • 多Agent → crewAI / AutoGen / 自建

    •

    "LangGraph 和 LangChain Agent 的区别？"

    → LangChain Agent 是预定义的 ReAct 循环

    → LangGraph 可以自定义任意执行图

    → LangGraph 适合复杂工作流，LangChain Agent 适合简单任务

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
"n">4.2 LangGraph —— 状态机 Agent
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

LangGraph 是 LangChain 生态中的 Agent 编排框架。
它的核心思想是用「有向图」来定义 Agent 的执行流程。

核心概念：
  - State: 贯穿整个流程的共享状态（TypedDict）
  - Node: 图中的一个执行节点（Python 函数）
  - Edge: 节点之间的连线（普通边 / 条件边）

什么时候用 LangGraph 而不是 LangChain Agent？
  - 需要自定义执行流程（不是简单的 ReAct 循环）
  - 需要在步骤间传递复杂状态
  - 需要并行执行、条件分支、循环等复杂控制流
  - 多 Agent 协作场景（第5章会用到）

安装依赖：
  pip install langgraph
"""

"k">class="h"># 定义状态类型
"k">class "f">AgentState(TypedDict):
    "k">class="s">"""LangGraph Agent 的状态定义。

    这是 LangGraph 最核心的概念。所有节点共享这个状态，
    节点通过读取/修改状态来协作完成任务。

    Annotated["b">list, add_messages] 表示消息列表使用
    追加模式（新消息添加到末尾，不会覆盖）。
    """
    messages: Annotated["b">list, add_messages]
    task_result: "b">str
    step_count: "b">int


"k">def "f">demo_langgraph_agent():
    "k">class="s">"""演示 LangGraph 状态机 Agent。

    实现一个简单的分析流程：
      输入问题 → LLM 分析 → 判断是否需要工具 → 输出结论

    LangGraph 让我们可以自定义 Agent 的执行图结构，
    而不是拘泥于固定的 ReAct 循环。
    """
    "k">if "k">not LANGGRAPH_AVAILABLE "k">or "k">not LANGCHAIN_AVAILABLE:
        "b">print("需要安装 langgraph 和 langchain 才能运行此演示。")
        "k">return

    "b">print("\n" + "=" * "n">60)
    "b">print("  LangGraph 状态机 Agent 演示")
    "b">print("=" * "n">60)

    llm = ChatOpenAI(
        model=os.getenv("LLM_MODEL", "gpt-4o-mini"),
        api_key=os.getenv("OPENAI_API_KEY"),
        base_url=os.getenv("OPENAI_BASE_URL"),
        temperature="n">0.5,
    )

    "k">class="h"># ===== 定义节点函数 =====
    "k">def "f">analyze_node(state: AgentState) -> AgentState:
        "k">class="s">"""分析节点：让 LLM 理解用户输入。"""
        "b">print("  [Node: analyze] 正在分析用户输入...")
        response = llm.invoke([
            ("system", "分析用户问题，提取关键信息。简洁回答。"),
            ("user", state["messages"][-"n">1].content),
        ])
        state["messages"].append(response)
        state["task_result"] = response.content
        state["step_count"] = state.get("step_count", "n">0) + "n">1
        "k">return state

    "k">def "f">should_continue(state: AgentState) -> "b">str:
        "k">class="s">"""条件判断：是否需要进一步处理？"""
        result = state["task_result"]
        "k">if "b">len(result) < "n">50:
            "b">print("  [Edge] 回答太短，需要补充详细分析")
            "k">return "elaborate"
        "b">print("  [Edge] 回答充分，结束流程")
        "k">return "end"

    "k">def "f">elaborate_node(state: AgentState) -> AgentState:
        "k">class="s">"""扩展节点：补充详细分析。"""
        "b">print("  [Node: elaborate] 正在补充详细分析...")
        response = llm.invoke([
            ("system", "请更详细地展开分析，提供更多上下文和细节。"),
            ("user", state["task_result"]),
        ])
        state["messages"].append(response)
        state["task_result"] = response.content
        state["step_count"] = state.get("step_count", "n">0) + "n">1
        "k">return state

    "k">class="h"># ===== 构建图 =====
    workflow = StateGraph(AgentState)

    "k">class="h"># 添加节点
    workflow.add_node("analyze", analyze_node)
    workflow.add_node("elaborate", elaborate_node)

    "k">class="h"># 设置入口点
    workflow.set_entry_point("analyze")

    "k">class="h"># 添加条件边：从 analyze 根据条件走向不同分支
    workflow.add_conditional_edges(
        "analyze",
        should_continue,
        {
            "elaborate": "elaborate",
            "end": END,
        },
    )

    "k">class="h"># elaborate 完成后结束
    workflow.add_edge("elaborate", END)

    "k">class="h"># 编译图
    app = workflow.compile()

    "k">class="s">"""
    图结构示意：

      [START]
         │
         ▼
    ┌──────────┐     条件判断      ┌───────────┐
    │  analyze  ├──────────────────→│ elaborate  │
    └─────┬─────┘  (太短)          └─────┬─────┘
          │ (足够)                       │
          ▼                              ▼
       [END]                          [END]

    这就是 LangGraph 的核心优势：
    用代码定义复杂的执行流程，而非硬编码在循环中。
    """

    "k">class="h"># 运行

    "b">print(f"\n👤 用户: {test_input}")

    initial_state = {
        "messages": [HumanMessage(content=test_input)],
        "task_result": "",
        "step_count": "n">0,
    }

    final_state = app.invoke(initial_state)
    "b">print(f"\n🤖 Agent: {final_state['task_result']}")
    "b">print(f"   总步骤数: {final_state['step_count']}")
