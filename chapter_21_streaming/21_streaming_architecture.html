    21.1 为什么 Agent 需要实时架构？

     

    传统「请求-响应」模式的问题：

     

    用户 → 发送消息 → [等待 30 秒...] → 收到完整回复

     

    Agent 执行过程中：

  • 用户不知道 Agent 在做什么
  • 用户发现方向不对，但只能等 Agent 完成再纠正
  • 每个步骤都是「黑盒」

    •  

    实时架构的目标：

     

    用户 → 发送消息 → 实时看到 Agent 的思考过程 →

         中途可以修正方向 → 最终得到结果

     

    核心能力：

  • 流式输出：LLM 每生成一个 token 就推送给用户
  • 中途介入：用户可以在 Agent 执行中途发送消息
  • 工具调用可见：用户能看到 Agent 在调什么工具
  • 可中断：用户可以随时停止 Agent 的任务

    •  

    这是 Claude Code h2A 机制（Ch8）和 Ch13 SSE/WebSocket 的

    「为什么」解释 —— 本章讲架构设计层面。

     

     

    21.2 事件驱动架构 (EDA) —— Agent 系统的骨架

     

    核心思想：

    传统架构：线性调用 → 阻塞等待 → 同步返回

    事件驱动：发布事件 → 异步处理 → 流式推送

     

    Agent 的关键事件类型：

     

      ┌─────────────────────────────────────────────┐
  │              Agent 事件体系                   │
  ├──────────────┬──────────────────────────────┤
  │ 事件类型       │         说明                 │
  ├──────────────┼──────────────────────────────┤
  │ user_input   │ 用户发送消息                   │
  │ thinking     │ Agent 思考中                   │
  │ token_stream │ LLM 流式输出的一个 token       │
  │ tool_call    │ Agent 决定调用工具              │
  │ tool_result  │ 工具执行完成                    │
  │ interrupt    │ 用户中途介入/停止              │
  │ error        │ 发生错误                        │
  │ done         │ Agent 任务完成                  │
  └──────────────┴──────────────────────────────┘

     

    关键组件：

    EventBus：事件总线，所有组件通过它通信

    → 解耦 LLM 调用、工具执行、UI 更新

     

    StateManager：状态管理器，唯一数据源

    → 记录 Agent 当前步骤、对话历史、工具执行状态

     

    StreamBridge：流式桥接

    → 将 LLM 的 streaming 输出桥接到前端

     

     

    21.3 EventBus 模式 —— Agent 架构的核心

     

    问题：Agent 的多个组件（LLM 调用 / 工具执行 / 前端 UI）

      怎么高效通信而不互相耦合？

     

    方案：EventBus —— 发布-订阅模式

     

      ┌──────────────────────────────────────────────────┐
  │                   EventBus                        │
  │                   (消息中枢)                       │
  │                                                    │
  │   pub/sub         pub/sub          pub/sub        │
  │   token_stream    tool_call         interrupt      │
  │      ↕               ↕                 ↕           │
  │  ┌────────┐    ┌──────────┐    ┌──────────┐     │
  │  │  LLM   │    │  Tool    │    │  User    │     │
  │  │ Service│    │ Executor │    │  Interface│     │
  │  └────────┘    └──────────┘    └──────────┘     │
  └──────────────────────────────────────────────────┘

     

    EventBus 的优势：

  • 解耦：LLM 不需要知道 UI 的存在
  • 可测试：可以 Mock EventBus 单独测每个组件
  • 可扩展：新组件只需订阅感兴趣的事件
  • 可观测：所有事件都可以记录（天然 Tracing）

    21.4 动态中断 (Dynamic Interrupt) —— Agent 交互的质变

     

    这是 Agent 从「自动化脚本」升级到「交互式伙伴」的关键能力。

     

    传统模式：提交任务 → 等完成 → 看结果 → （不满意）→ 重新提交

    中断模式：提交任务 → 观察进行中 → 途中发现方向不对 → 立即纠正

     

    架构实现：

  • EventBus 监听 interrupt 事件
  • Agent 循环每步检查 should_interrupt()
  • 如果中断 → 暂停执行 → 等待用户新指令
  • 用户发送新指令 → resume 事件 → 继续执行

    •  

    和 Claude Code h2A 的关系（Ch8）：

    h2A 是「双缓冲异步队列」

    EventBus 模式是 h2A 的「通用化实现」

     

    关键在于中断的「粒度」：

  • 太粗：等当前工具调完才知道 → 延迟大
  • 太细：每收到一个 token 就检查 → 开销大
  • 最优：每个 Agent 步骤结束时检查（Claude Code 的做法）

    •  

    可中断 Agent —— 演示动态中断的完整流程。

     

    结合 EventBus + StateManager + 模拟 LLM。

     

    用户请求中断。

     

    用户恢复执行（可带新指令）。

    21.5 背压控制 (Backpressure) —— 生产环境的隐性需求

     

    场景：

    LLM 的 token 生成速度快于前端的渲染速度

    → 前端缓冲区满了 → 如果不处理 → 丢数据或 OOM

     

    解决方案：

     

  • 缓冲区控制
    •  前端维护一个固定大小的缓冲队列
 满了就暂停 SSE 流

     

  • 自适应降频
    •  如果前端处理变慢 → 减少推送频率
 从「每 token 推送」降级到「每 5 token 推送」

     

  • 生产者-消费者模式
    •  生产者（LLM）→ 队列 → 消费者（前端）
 队列有容量上限，满了暂停生产者

     

    这就是 Claude Code h2A 的「智能背压控制」

    （Ch8 提到：吞吐量 > 10,000 消息/秒 + 智能背压）。

     

    背压控制器 —— 防止生产者速度超过消费者。

     

    生产者-消费者队列，内置流控机制。

     

    生产者：发布事件。

     

        Returns:
        是否成功入队。

     

    消费者：取出事件。

     

    背压统计。

    21.6 本章总结

     

    核心要点回顾：

     

  • Event-Driven = Agent 实时交互的基础
  • EventBus 解耦各组件
  • StateManager 做唯一数据源
  • 事件驱动使中断、流式输出、工具调用可见成为可能

class BackpressureController:
def __init__(self, max_size: int = 100):
        self.queue = asyncio.Queue(maxsize=max_size)
        self.dropped = 0  class=class="s">"h"># 丢弃的事件数
        self.total = 0

    async def produce(self, event: AgentEvent) -> bool:
self.total += 1
        try:
            self.queue.put_nowait(event)
            return True
        except asyncio.QueueFull:
            self.dropped += 1
            return False

    async def consume(self) -> AgentEvent:
return await self.queue.get()

    def stats(self) -> dict:
return {
            class="s">"queue_size": self.queue.qsize(),
            class="s">"total_produced": self.total,
            class="s">"dropped": self.dropped,
            class="s">"drop_rate": self.dropped / max(self.total, 1),
        }
