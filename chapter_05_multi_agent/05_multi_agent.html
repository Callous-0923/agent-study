=========================================

 

 

 

 

5.1 为什么需要 Multi-Agent？

 

一个类比：

 

单 Agent = 一个全栈工程师

  • 什么都能做，但每项都不够专业
  • 上下文太长，容易分心

    •  

    多 Agent = 一个专业团队

  • 每人各司其职（数据分析师 + 程序员 + 文案）
  • 通过分工协作完成复杂任务
  • 每个 Agent 的 prompt 可以高度专业化

    •  

    核心优势：

  • 分工明确：每个 Agent 专注自己的领域
  • 并行执行：多个 Agent 可以同时工作
  • 错误隔离：一个 Agent 出错不影响整体
  • 专业化 prompt：每个 Agent 可以用领域最优提示词

    •  

    经典场景：

  • 软件开发生命周期（需求 → 设计 → 编码 → 测试 → 部署）
  • 内容生产流水线（调研 → 写作 → 编辑 → 发布）
  • 投资分析（宏观分析 → 行业分析 → 个股分析 → 风险评估）

    •  

    三种经典架构：

     

    ┌──────────────┬───────────────────────────────────────┐
│    架构       │               核心思想                  │
├──────────────┼───────────────────────────────────────┤
│ 协作式        │ 多个 Agent 平等对话，共同决策            │
│ (Horizontal) │ 典型：AutoGen 的 GroupChat              │
├──────────────┼───────────────────────────────────────┤
│ 分层式        │ 上级 Agent 委派任务给下级 Agent          │
│ (Vertical)   │ 典型：crewAI 的 Crew + Task 模型        │
├──────────────┼───────────────────────────────────────┤
│ 竞争式        │ 多个 Agent 各自生成方案，择优选用         │
│ (Competitive)│ 典型：辩论式 Agent / Self-Play          │
└──────────────┴───────────────────────────────────────┘

    5.3 crewAI 风格的角色-Crew 模型

     

    crewAI 采用「Crew = Agents + Tasks」模型：

  • 定义一系列 Agent（角色）
  • 定义一系列 Task（任务）
  • 将任务分配给 Agent 组成 Crew
  • Crew 自动协调执行

    •  

    虽然我们不直接安装 crewAI（避免依赖过重），

    但可以用 LangChain 模拟其核心概念来理解其设计思想。

  2. Agent 之间通过任务(task)来协作
  3. 可以设置 allow_delegation 允许 Agent 把任务委派给其他 Agent

    5.4 Multi-Agent 设计原则（面试高频！）

     

  • 职责单一原则
  • 每个 Agent 只做一件事，做到极致
  • 不要给一个 Agent 塞太多职责

    •  

  • 通信标准化
  • Agent 之间用结构化格式通信（JSON/XML）
  • 避免自然语言的模糊性

    •  

  • 错误隔离
  • 一个 Agent 失败不应导致整个系统崩溃
  • 设置重试机制和降级策略

    •  

  • 可观测性
  • 记录每个 Agent 的输入/输出/耗时
  • 便于调试和优化

    •  

  • 成本控制
  • Multi-Agent 会成倍增加 API 调用量
  • 用缓存、batching、模型分层来降低成本

    •  

    面试常问：

    "Multi-Agent 比单 Agent 好在哪里？什么时候不该用？"

     

    好处：

  • 分工专业化，输出质量更高
  • 复杂任务可以并行处理
  • 系统更模块化、易维护

    •  

    不适用场景：

  • 简单任务（杀鸡用牛刀）
  • 成本敏感场景（多倍 API 调用）
  • Agent 间协调开销 > 收益的场景

    •  

    5.5 本章总结

     

    核心要点回顾：

     

  • Multi-Agent = 多角色协作
  • 协作式：平等对话，共同决策
  • 分层式：上级委派，下级执行
  • 竞争式：多方生成，择优选用

    •  

  • 通信机制
  • 共享状态（LangGraph 的方式）
  • 消息传递（AutoGen 的方式）
  • 任务委派（crewAI 的方式）

    •  

  • 实践框架
  • LangGraph：灵活构建自定义 Multi-Agent 图
  • crewAI：开箱即用的角色-任务模型
  • AutoGen：对话驱动的 Multi-Agent

    •  

  • 设计原则
  • 职责单一、通信标准化、错误隔离、可观测、控成本

    •  

    面试速记：

    → 说明架构设计（用什么模式、为什么）

    → 展示效果提升（效率/质量提升多少）

class ContentTeamState(TypedDict):
    class=class="s">"s">class="s">"""内容创作团队的状态。

    所有 Agent 节点共享此状态，通过读写状态来协作。
    这是 LangGraph 多 Agent 系统的核心：
    状态是 Agent 之间的「共享白板」。

    messages: Annotated[list, add_messages]
    task: str                      class=class="s">"h"># 原始任务
    draft: str                     class=class="s">"h"># Writer 的草稿
    review: str                    class=class="s">"h"># Reviewer 的审核意见
    iteration: int                 class=class="s">"h"># 当前迭代轮次
    approved: bool                 class=class="s">"h"># 是否通过审核
    max_iterations: int            class=class="s">"h"># 最大迭代次数


WRITER_SYSTEM_PROMPT = class=class="s">"s">class="s">"""你是一位专业的内容创作者。
风格要求：
- 语言简洁有力
- 逻辑清晰，论据充分
- 面向技术读者的科普风格
- 字数在 300-500 字之间class="s">"""

REVIEWER_SYSTEM_PROMPT = class=class="s">"s">class="s">"""你是一位严格的内容审核编辑。
审查标准：
- 准确性：内容是否有事实性错误？
- 逻辑性：论述是否严密？
- 可读性：是否通俗易懂？
- 完整性：是否覆盖了任务要求的所有方面？

返回格式：
{class="s">"score": 1-10, class="s">"issues": [class="s">"问题1", class="s">"问题2"], 
 class="s">"approved": true/false, class="s">"suggestions": class="s">"改进建议"}class="s">"""


def build_content_team(llm):
    class=class="s">"s">class="s">"""构建内容创作双 Agent 团队。

    Args:
        llm: LangChain ChatOpenAI 实例。

    Returns:
        编译后的 LangGraph 应用。
    class="s">"""

    def writer_node(state: ContentTeamState) -> ContentTeamState:
        class=class="s">"s">class="s">""class="s">"Writer Agent：创作内容或根据审核意见修改。"class="s">""
        iteration = state.get(class="s">"iteration", 0)
        review = state.get(class="s">"review", class="s">"")

        if review and iteration > 0:
            prompt = (
                fclass="s">"原始任务: {state[class='s'>'task']}\n\n"
                fclass="s">"上次草稿: {state[class='s'>'draft']}\n\n"
                fclass="s">"审核意见: {review}\n\n"
                fclass="s">"请根据审核意见修改草稿。"
            )
            print(fclass="s">"  ✏️ [Writer] 第{iteration}轮修改...")
        else:
            prompt = fclass="s">"任务: {state[class='s'>'task']}\n请撰写内容。"
            print(fclass="s">"  ✏️ [Writer] 开始创作...")

        response = llm.invoke([
            (class="s">"system", WRITER_SYSTEM_PROMPT),
            (class="s">"user", prompt),
        ])
        state[class="s">"draft"] = response.content
        return state

    def reviewer_node(state: ContentTeamState) -> ContentTeamState:
        class=class="s">"s">class="s">""class="s">"Reviewer Agent：审核草稿质量。"class="s">""
        print(fclass="s">"  🔍 [Reviewer] 正在审核...")

        prompt = (
            fclass="s">"任务要求: {state[class='s'>'task']}\n\n"
            fclass="s">"待审核草稿:\n{state[class='s'>'draft']}\n\n"
            fclass="s">"请审核并评分。返回 JSON 格式。"
        )
        response = llm.invoke([
            (class="s">"system", REVIEWER_SYSTEM_PROMPT),
            (class="s">"user", prompt),
        ])

        try:
            result = json.loads(response.content)
        except json.JSONDecodeError:
            result = {class="s">"score": 5, class="s">"issues": [class="s">"无法解析审核结果"],
                       class="s">"approved": True, class="s">"suggestions": class="s">""}

        state[class="s">"review"] = result.get(class="s">"suggestions", class="s">"")
        state[class="s">"approved"] = result.get(class="s">"approved", False)
        print(fclass="s">"    评分: {result.get(class='s'>'score', class='s'>'N/A')}/10 | "
              fclass="s">"通过: {state[class='s'>'approved']} | "
              fclass="s">"问题: {result.get(class='s'>'issues', [])}")
        return state

    def should_continue(state: ContentTeamState) -> Literal[class="s">"writer", class="s">"end"]:
        class=class="s">"s">class="s">"""决定是否继续迭代。

        Returns:
            class='s'>'writer': 需要 Writer 修改
            class='s'>'end': 审核通过或达到最大迭代次数
        class="s">"""
        iteration = state.get(class="s">"iteration", 0)
        max_iter = state.get(class="s">"max_iterations", 3)
        approved = state.get(class="s">"approved", False)

        if approved:
            print(fclass="s">"  ✅ 审核通过！总迭代 {iteration + 1} 轮")
            return class="s">"end"

        if iteration >= max_iter - 1:
            print(fclass="s">"  ⚠️ 达到最大迭代次数 {max_iter}，强制结束")
            return class="s">"end"

        state[class="s">"iteration"] = iteration + 1
        return class="s">"writer"

    class=class="s">"h"># 构建图
    workflow = StateGraph(ContentTeamState)

    workflow.add_node(class="s">"writer", writer_node)
    workflow.add_node(class="s">"reviewer", reviewer_node)

    workflow.set_entry_point(class="s">"writer")
    workflow.add_edge(class="s">"writer", class="s">"reviewer")
    workflow.add_conditional_edges(
        class="s">"reviewer",
        should_continue,
        {class="s">"writer": class="s">"writer", class="s">"end": END},
    )

    return workflow.compile()

class RoleBasedAgent:
    class=class="s">"s">class="s">"""模拟 crewAI 风格的角色 Agent。

    crewAI 的核心设计理念：
      1. 每个 Agent 有明确的角色(role)、目标(goal)、背景(backstory)
      2. Agent 之间通过任务(task)来协作
      3. 可以设置 allow_delegation 允许 Agent 把任务委派给其他 Agent
    class="s">"""

    def __init__(self, name: str, role: str, goal: str, backstory: str,
                 llm=None):
        class=class="s">"s">class="s">"""初始化角色 Agent。

        Args:
            name: Agent 名称。
            role: 角色描述（如 class="s">"数据分析师"）。
            goal: 目标描述。
            backstory: 背景故事（帮助 LLM 理解角色定位）。
            llm: LangChain LLM 实例。
        class="s">"""
        self.name = name
        self.role = role
        self.goal = goal
        self.backstory = backstory
        self.llm = llm

    def execute(self, task: str) -> str:
        class=class="s">"s">class="s">"""执行分配给此 Agent 的任务。

        Args:
            task: 任务描述。

        Returns:
            执行结果。
        class="s">"""
        system_prompt = (
            fclass="s">"你是 {self.name}，一名 {self.role}。\n"
            fclass="s">"你的目标: {self.goal}\n"
            fclass="s">"你的背景: {self.backstory}"
        )
        if self.llm is None:
            return fclass="s">"[{self.name}] 模拟输出：已完成任务 class='s'>'{task}'"
        response = self.llm.invoke([
            (class="s">"system", system_prompt),
            (class="s">"user", task),
        ])
        return response.content

5.4 Multi-Agent 设计原则（面试高频！）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 职责单一原则
   - 每个 Agent 只做一件事，做到极致
   - 不要给一个 Agent 塞太多职责

2. 通信标准化
   - Agent 之间用结构化格式通信（JSON/XML）
   - 避免自然语言的模糊性

3. 错误隔离
   - 一个 Agent 失败不应导致整个系统崩溃
   - 设置重试机制和降级策略

4. 可观测性
   - 记录每个 Agent 的输入/输出/耗时
   - 便于调试和优化

5. 成本控制
   - Multi-Agent 会成倍增加 API 调用量
   - 用缓存、batching、模型分层来降低成本

面试常问：
  class="s">"Multi-Agent 比单 Agent 好在哪里？什么时候不该用？"

  好处：
    - 分工专业化，输出质量更高
    - 复杂任务可以并行处理
    - 系统更模块化、易维护

  不适用场景：
    - 简单任务（杀鸡用牛刀）
    - 成本敏感场景（多倍 API 调用）
    - Agent 间协调开销 > 收益的场景
